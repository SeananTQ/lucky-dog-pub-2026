# Steam 商店页提交审核复盘

本文档记录 `Lucky Dog Rise` 首次提交 Steam 商店页审核前后的关键决策、已完成内容、素材策略与后续待办。

当前目标已经从“补齐清单”切换为“等待 Valve 审批，并准备审核通过后的 Coming Soon 发布与愿望单积累”。

## 产品定位

`Lucky Dog Rise` 当前商店页围绕以下卖点组织：

- 稀缺的小狗桌宠：竞品中猫、鱼、小虫等桌面伙伴较多，小狗题材相对少见。
- 幸运感：小狗陪伴玩家工作、休息和游玩，并通过幸运点数、扑克和奖励循环强化“带来好运”的情绪。
- 输入节奏反馈：小狗不是单纯重复动作，而是会根据玩家打字、点击和停顿的节奏切换反应。
- 双模式结构：桌宠模式负责陪伴与轻量成长，打牌模式负责短暂消遣和奖励收益。
- 更丰富的装扮：不只装扮小狗，也能装扮玩家角色和背景。

公开表达上优先使用 `Lucky Points / 幸运点数`，减少“筹码”一词带来的博彩联想。游戏内 UI 仍可按当前实现显示 `Chips`，后续本地化时再统一评估。

## 内容调查结论

内容调查按实际玩法保守填写：

- 游戏包含视频扑克式玩法和概率奖励体验。
- 游戏不包含真实货币赌博。
- 游戏内奖励不可兑换现金或现实奖品。
- 没有把赌博作为核心营销卖点。
- 酒精/烟草相关评级风险来自饮品类装饰物，而不是饮酒玩法。
- 游戏不包含运行时生成式 AI 系统。

AI 相关口径：

- 可以使用机器翻译或 AI 辅助翻译作为短 UI、道具名、商店文案和 Steam 库存元数据的草稿工具。
- 所有发布给玩家的文字都应由主人人工审阅、编辑并确认。
- 游戏不包含运行时生成式 AI、不生成玩法内容、不生成美术或音频，也不向玩家提供 AI 系统。

审核后生成的地区评级结果可接受：

- Brazil：`10`
- Germany：`6`，原因包含 `Alkohol / Tabak`

## 语言策略

本次提交采用分离策略：

- 商店页本地化文本：可以保留英文、简体中文、繁体中文、日文。
- 游戏内支持语言：仅声明 `English - Interface`。

原因：

- 商店页本地化和游戏内支持语言是两类承诺。
- 当前游戏 UI 为英文硬编码，尚未接入正式本地化系统。
- 提前在支持语言表中勾选中文或日文，容易让玩家误以为游戏内 UI 已经支持对应语言。

后续当本地化系统完成后，再补充：

- `Simplified Chinese - Interface`
- `Traditional Chinese - Interface`
- `Japanese - Interface`

公开前必须再次核对 Steam 页面右侧语言表：

- 如果游戏内 UI 仍未完成中文、繁中和日文本地化，商店页语言表不应声明这些语言支持 `Interface`。
- 商店页“描述文本的本地化”和游戏内“支持语言”是两类承诺，需要分开处理。

## 素材策略

### 总原则

本次审核素材目标是“真实、可读、能过审”，不是一次性做到最终商业美术版本。

素材不追求 95 分终稿，而是先达到 70 分可提交版本：

- Logo 清楚。
- 小狗可爱。
- 桌宠属性明确。
- 扑克牌元素有轻度暗示。
- 不使用误导玩家的概念图当截图。

后续审核通过或页面公开后，可以继续替换更精修的素材。

### Logo

Logo 采用：

- `Lucky Dog` 作为主体品牌。
- `Rise` 作为副标题徽章。
- 视觉理解接近 `Lucky Dog: Rise`，但商店标题仍写作 `Lucky Dog Rise`。

字体策略：

- 不把 AI 草图中的字形视为可直接使用的字体。
- 推荐以可商用字体作为骨架，例如 `Fredoka`、`Baloo 2`、`Nunito`、`Coiny`、`Lilita One`。
- 在 Illustrator 中转轮廓后进行手工调整，形成专属字标。

### 商店页背景

`store_page_background` 实际展示机制与最初预期不同：

- Steam 会放大、裁切、压暗并模糊背景。
- 画面下方和中心容易被前景截图区、信息栏和遮罩覆盖。
- 玩家真正容易看到的是左右两侧外露区域。

最终方向调整为：

- 不再依赖真实打牌画面的完整构图。
- 使用大尺寸、低对比度的扑克牌元素作为背景暗示。
- 让左侧 `A` 与右侧 `K` 在页面外露区域可被隐约看见。
- 中间保持弱化，不与前景内容抢视觉。

该方案能让玩家潜意识知道游戏含有扑克牌玩法，但不会把首屏误导成纯扑克或赌场游戏。

### Header Capsule

Header Capsule 采用：

- 左侧 Logo。
- 右侧小狗。
- 底部伪造任务栏。
- 浅色背景与轻微扑克牌/爪印纹理。

任务栏不使用真实 Windows 截图，而是伪造的游戏化任务栏：

- 减少系统图标、时间、托盘状态等噪音。
- 强化“桌面宠物”印象。
- 避免真实系统 UI 带来的廉价感和不可控信息。

### Vertical Capsule

竖版宣传图使用纸箱作为小狗支撑物。

原因：

- 竖版图横向空间不足，难以清楚表现任务栏。
- 小狗舌头和爪子需要依附在某个物体上。
- 纸箱可以表达“收养一只小狗”的情绪，与桌宠陪伴定位一致。

纸箱与任务栏并不冲突：

- 纸箱：表达“带回家/收养一只小狗”。
- 任务栏：表达“小狗住在电脑桌面上”。
- 扑克牌纹理：表达“幸运与打牌玩法”。

### 图标

快捷方式图标：

- 使用 `windows_icon.ico`
- 由 `512x512` PNG 通过 ImageMagick 自动生成多尺寸 ICO：

```powershell
magick lucky_dog_icon_512.png -define icon:auto-resize=256,128,64,48,32,16 windows_icon.ico
```

该命令会从单张 512 图自动缩放生成多个尺寸并打包进 ICO，不会从文件夹中读取已有小图。

应用图标：

- Steam 要求 `184x184 JPG`。
- 建议命名为 `steam_app_icon_184.jpg`，避免误以为是 macOS 图标。

ICNS：

- 仅 macOS 构建需要。
- 当前如果仅发布 Windows 版，可以暂时跳过。

## 截图策略

截图已上传并用于本次审核。

推荐截图组合为：

1. 桌宠模式主图：小狗在干净桌面上陪伴玩家。
2. 桌宠反馈图：展示小狗反应、气泡或幸运点数变化。
3. 扑克模式图：展示狗、牌桌、手牌、玩家手和奖励信息。
4. 盲盒或奖励图：展示奖励循环。
5. 背包/装扮图：展示收集和自定义。

上传截图时，“此截图适合所有年龄段人士吗？”的判断口径：

- 桌宠截图、背包截图、盲盒截图和普通打牌截图均可选“是”。
- 普通扑克牌画面不等于不适合全年龄。
- 如果未来截图明显突出酒精、成人内容、强暴力、真实货币赌博或其他敏感内容，再单独评估。

## 发行日期与折扣

发行日期：

- Steam 后台不允许选择部分周末日期。
- 预定发行日期可以在正式发布前调整。
- 该日期不会自动触发正式发行，正式发行仍需要主人手动发布。

当前建议：

- 不要选择过近日期，避免给开发排期制造不必要压力。
- 如果后台必须填具体日期，优先选择工作日。
- 对玩家展示时尽量使用月份或季度级别的窗口，减少频繁改期风险。

首发折扣：

- 本次 Coming Soon 审核阶段无需设置首发折扣。
- 建议接近正式发售前，再根据定价、愿望单数量和竞品表现决定。
- 当前倾向：首发折扣可先不设；若需要，后续考虑 `10% / 7 天` 作为保守方案。

## Playtest 策略

Steam Playtest 不属于本次商店页提交审核的主线。

后续如需 Playtest，需要：

- 主游戏商店页公开，便于玩家看到报名入口。
- 创建关联的 Playtest App。
- 准备 Playtest 的库资产、社区资产和可玩的测试 build。
- 在主游戏商店页 Special Settings 中启用报名入口。

建议顺序：

1. 商店页审核通过。
2. 手动发布为 Coming Soon。
3. 开始积累愿望单。
4. 准备稳定测试 build。
5. 再开启 Playtest。

Playtest 适合验证：

- 透明窗口与点击穿透兼容性。
- 桌宠是否干扰玩家日常使用。
- 全局键鼠统计稳定性。
- 多分辨率和多显示器表现。
- 打牌模式理解成本。
- 幸运点数、盲盒和装扮节奏。

## 审核通过后的下一步

审核通过后，主人需要手动决定是否公开 Coming Soon 页面。

建议操作顺序：

1. 阅读 Valve 审核反馈。
2. 如果需要修改，按反馈修正后重新提交。
3. 如果审核通过，先做一次最终页面预览。
4. 确认公开文案、截图、胶囊图、年龄与语言承诺无误。
5. 手动发布为 `Coming Soon / 即将推出`。
6. 开始观察愿望单、地区流量和标签表现。

公开后可继续优化：

- 精修 Header、Main、Small、Vertical Capsule。
- 制作正式预告片。
- 视转化表现再决定是否制作 About This Game 用的 WebM/GIF；当前不是高优先级。
- 完成本地化系统后补充中文、日文支持语言。
- 准备 Steam Playtest。
- 设计 Steam 成就和云存档接入。

## 后续待办
