# 盲盒系统设计

## 设计目标

盲盒系统用于承接玩家的长期收集目标。玩家可以通过开盲盒、活动领取、签到领奖、调试发奖等不同来源获得小狗、头饰、眼镜、桌布、背景、卡背等装扮道具。

从底层设计看，本系统应被视为“发奖系统”，普通盲盒只是其中一种展示形式。

Steam 版本的正式库存应优先接入 Steam Inventory Service。游戏本地存档负责保存装备状态、New 标记、待揭晓状态、开盒演出进度和调试数据；正式道具库存不应只依赖本地 JSON 存档。

系统设计需要满足以下目标：

1. 玩家视角下，开盒流程清晰，盲盒升级表演有期待感。
2. 策划视角下，奖励来源、展示方式、品质概率、活动发奖可以分别配置。
3. Steam 版本中，正式道具尽量由 Steam Inventory 托管，降低玩家本地作弊抽穿图鉴的风险。
4. 开盒过程中不能穿帮。玩家在动画揭晓前，不应从游戏内背包、New 标记、图鉴、成就、任务或其他反馈中提前得知最终奖励。

## Steam Inventory 定位

Steam Inventory Service 用于托管玩家正式库存。Steam 版本中，正式道具、盲盒箱子、开箱券、活动券、签到进度令牌等都应优先设计为 Steam Inventory 物品。

游戏客户端负责：

1. 展示背包和装备界面。
2. 播放开盒动画。
3. 根据 Steam 返回的结果生成本地演出路径。
4. 保存待揭晓状态和演出进度。
5. 在奖励揭晓后触发 New 标记、图鉴、成就、任务进度等本地反馈。

Steam 负责：

1. 持久化正式库存。
2. 校验交换材料。
3. 发放 generator、bundle 或 item 的最终结果。
4. 支持活动、促销、掉落、购买、交换等库存规则。

在没有自建服务器的前提下，正式发奖逻辑应尽量使用 Steam 已支持的规则组合，而不是由本地客户端直接决定最终奖励。

## 通用发奖系统

通用发奖系统负责处理“玩家通过某个来源获得某个奖励”的流程。它不应绑定某一种 UI 表现。

发奖来源可以包括：

1. 玩家使用盲盒箱子或开箱券开启普通盲盒。
2. 玩家通过 LinkTree 或活动入口领取活动奖励。
3. 玩家通过签到、定期领取或进度令牌领取奖励。
4. 调试功能直接发放测试道具。
5. 后续版本中的成就奖励、每日登录奖励、兑换码奖励、DLC 奖励等。

发奖展示方式可以包括：

1. 普通盲盒升级表演。
2. 直接展示奖励图标。
3. 多奖励列表展示。
4. 调试模式直接入账。

普通盲盒、活动盲盒、签到奖励都应共用“发奖结果进入待揭晓或已揭晓流程”的底层规则。

## 普通盲盒

普通盲盒面向玩家的日常收集循环。

Steam 版本中，普通盲盒不应由本地程序直接抽出最终奖励。推荐流程为：

1. 玩家拥有一个 Steam Inventory 中的盲盒箱子或开箱券。
2. 玩家点击开盲盒。
3. 客户端调用 Steam Inventory 的交换流程，消耗箱子或开箱券。
4. Steam 根据配置好的 generator 返回最终道具。
5. 客户端把该结果记录为待揭晓库存项。
6. 客户端根据最终道具品质生成开盒演出路径。
7. 玩家推进盲盒升级动画。
8. 最终奖励图标展示后，该物品在游戏内转为已揭晓，并触发 New 标记和相关反馈。

从 Steam 视角看，交换成功后物品已经进入玩家正式库存。  
从游戏内 UI 视角看，该物品在揭晓前仍处于待揭晓状态，不应显示在背包中。

## 活动盲盒

活动盲盒面向限时活动、外部入口或特殊领取场景。

例如圣诞活动盲盒可以包含 5 个 Special1 品质道具，且每个道具每名玩家只期望获得 1 个。玩家通过 LinkTree 点击活动网站后获得一次领取机会。领取时系统直接展示奖品图标，不播放普通盲盒的三次升级表演。

活动盲盒通常具有以下特点：

1. 不消耗筹码。
2. 不播放盲盒升级表演。
3. 可以固定奖励品质，例如只从 Special1 品质池中发奖。
4. 可以只包含活动套装物品。
5. 可以限制领取次数、活动有效期或领取条件。
6. 可以直接进入奖励展示和收起流程。

活动盲盒不应硬塞进普通盲盒的三次升级规则中。它应作为通用发奖系统的一种来源存在。

## 签到与顺序领奖

Steam Inventory 没有直接的“签到格子”专用系统，但可以通过 Steam 库存物品组合实现顺序领奖。

推荐设计是：

1. 玩家定期获得一个领取券。
2. 玩家库存中持有一个进度令牌，表示当前处于第几个格子。
3. 玩家点击领奖时，Steam 消耗领取券和当前进度令牌。
4. Steam 发放当前格奖励，并同时发放下一个进度令牌。
5. 当进度达到最后一格后，后续继续发放最后一格奖励，并保留最后一格进度令牌。

这种设计可以让 Steam 校验领取材料，避免完全依赖本地存档控制顺序。

## 玩家视角规则

玩家购买或获得普通盲盒后，进入盲盒开启流程。

开盒流程开始时，画面中显示一个盲盒。盲盒初始品质不一定总是普通品质，也可能直接以较高品质出现。较高初始品质用于制造“买到高级盲盒”的惊喜。

玩家可以点击盲盒推进开盒流程。每次点击后，盲盒会在原地跳跃，并在落下时展示本次是否升级。

每个普通盲盒最多拥有三次点击升级机会。每次升级幅度可能为 0 到 2 级：

1. 升级幅度为 0：盲盒品质不变。
2. 升级幅度为 1：盲盒提升一个品质。
3. 升级幅度为 2：盲盒连续提升两个品质。

盲盒最高品质为传说。当盲盒在三次点击结束前已经达到传说品质时，可以提前结束升级流程，并进入奖励展示阶段。

升级流程结束后，盲盒消失，画面展示最终获得的道具图标。玩家可以主动点击道具图标将其收起，也可以等待数秒后由系统自动收起。

## 品质设计

普通盲盒使用四个主要品质：

1. 普通
2. 稀有
3. 史诗
4. 传说

活动奖励可以使用普通盲盒品质，也可以使用 Special1、Special2 等特殊品质。

品质承担两个作用：

1. 表示玩家最终获得奖励的稀有度。
2. 表示开盒演出过程中盲盒当前所处的展示状态。

从玩家视角看，盲盒品质像是在点击过程中逐步变化。  
从系统视角看，最终奖励已经由 Steam Inventory 返回，盲盒升级过程是根据最终品质反推出来的演出。

## Steam 奖池结构

普通盲盒的“先随机品质，再在品质内随机物品”不应设计成多次 API 调用，而应设计成 Steam ItemDef 中的 generator 嵌套。

推荐结构为：

1. 盲盒箱子或开箱券是 Steam Inventory 中的普通 item。
2. 主 generator 负责按品质权重随机到某个品质 generator。
3. 品质 generator 负责在该品质下按权重随机到具体道具。
4. Steam 展开 generator 链后，最终发放一个具体道具。

客户端开盒时只发起一次交换流程。Steam 完成 generator 展开后返回最终结果。

这种结构可以保留策划需要的品质概率控制，也能让客户端根据最终品质反推开盒动画。

## 初始物品与奖池版本

由于 Steam server-less generator 不支持按玩家已有库存动态剔除候选项，普通盲盒池不应包含策划认为“抽到重复会明显不舒服”的初始物品。

初始物品应作为新玩家默认拥有内容，不进入普通盲盒池。

活动套装物品也不应强行进入普通盲盒池。它们应通过活动发奖、顺序领奖或活动专用 generator 处理。

对于已经发布过的 Steam 奖池，不建议直接修改旧箱子的掉落内容。更稳妥的做法是创建新的奖池版本：

1. 旧箱子继续使用旧奖池。
2. 新版本玩家获得的新箱子使用新奖池。
3. 如果某个旧盲盒物品被改为初始物品，应从新奖池中移除，而不是直接暗改旧奖池。

这样可以减少玩家对已持有箱子内容变化的争议。

## 物品获取类型

Item 表不应只依赖 `IsUnique` 和盲盒组 ID 表达物品来源。

建议增加一个“物品获取类型”字段，用于表达该物品主要从哪里获得。

推荐枚举含义：

1. 初始拥有：新玩家默认拥有，不进入普通盲盒。
2. 普通盲盒：进入普通盲盒奖池。
3. 活动奖励：通过活动、LinkTree、签到或其他活动入口获得。
4. 已下架：旧玩家已有则保留，新玩家不再获得。
5. 调试专用：仅用于开发和调试。

`IsUnique` 仍然可以用于本地背包显示、堆叠规则和装备逻辑，但不应被视为 Steam generator 中“自动剔除重复物品”的规则。

## 待揭晓库存项

待揭晓库存项是 Steam 版本盲盒系统的核心状态。

当 Steam 开箱交换成功后，最终道具已经进入玩家 Steam 库存。但在开盒动画揭晓前，游戏内不应把该物品当作已揭晓奖励处理。

待揭晓库存项与国家或地区隐藏不是同一类概念：

1. 国家或地区隐藏用于控制某些物品在特定地区是否可见、可抽或是否需要安全替换。
2. 待揭晓用于控制一个已经由 Steam 发放的物品，在开盒动画完成前是否能被游戏内系统看见。

在待揭晓期间，该物品不应出现在以下系统中：

1. 背包列表。
2. New 标记。
3. 自动装备逻辑。
4. 图鉴点亮。
5. 成就解锁。
6. 任务进度。
7. 活动收集进度。
8. 稀有物品播报。
9. 红点提示。
10. 玩家已经看见奖励的统计埋点。

只有当最终奖励图标正式展示后，该物品才从待揭晓状态转为已揭晓状态。

## 开盒演出生成规则

开盒演出根据最终奖励品质反推生成。

Steam 返回最终道具后，客户端通过本地 Item 表查到该道具品质，再从开盒演出路径配置中选择一条路径。

演出路径包含：

1. 初始盲盒品质。
2. 第一次点击后的升级幅度。
3. 第二次点击后的升级幅度。
4. 第三次点击后的升级幅度。
5. 是否允许在达到传说品质后提前结束。

演出路径必须保证最终展示品质与最终奖励品质一致。

例如最终奖励品质为传说时，系统可以生成以下演出：

1. 初始为普通。
2. 第一次点击不升级。
3. 第二次点击提升两个品质，变为史诗。
4. 第三次点击提升一个品质，变为传说。

也可以生成另一种演出：

1. 初始为稀有。
2. 第一次点击不升级。
3. 第二次点击提升两个品质，变为传说。
4. 因为已经达到传说，提前进入奖励展示阶段。

从玩家视角看，这些过程像是点击带来了升级结果。  
从系统视角看，这些过程是根据 Steam 返回的最终结果生成的表演剧本。

## 中断保护

中断保护的核心原则是“不穿帮”。玩家在看到最终奖励之前，不应该从游戏内任何系统提前得知最终奖励。

普通盲盒的推荐流程为：

1. 玩家点击开盲盒。
2. 客户端请求 Steam 执行交换。
3. Steam 成功返回最终道具。
4. 客户端保存待揭晓记录。
5. 客户端生成并保存开盒演出路径。
6. 玩家推进开盒演出。
7. 最终奖励图标首次展示。
8. 待揭晓物品转为已揭晓。
9. 系统触发 New 标记、图鉴、成就、任务、红点等获得型反馈。
10. 玩家点击奖励图标或等待自动收起后，开盒流程结束。

如果玩家在开盒过程中中断游戏，下次进入游戏后，该未完成盲盒仍然保留。

未完成盲盒会占用开盲盒入口。玩家再次点击开盲盒按钮时，系统不会购买或开启新的盲盒，而是继续打开之前中断的盲盒流程。

如果玩家中断时尚未看到最终奖励，则恢复到中断前的开盒进度，让玩家继续完成剩余点击和升级表演。

如果玩家中断时已经看到最终奖励图标，则该奖励已经可以在游戏内转为已揭晓。再次进入时可以恢复奖励展示界面，也可以视 UI 需要直接结束开盒流程，但不能丢失奖励。

中断保护需要保存以下信息：

1. 发奖来源 ID。
2. Steam 物品定义 ID。
3. Steam 物品实例 ID。
4. 最终奖励对应的本地 Item ID。
5. 最终奖励品质。
6. 开盒演出路径。
7. 当前演出步骤。
8. 是否已经展示最终奖励。
9. 是否已经完成已揭晓转换。

如果 Steam 库存和本地待揭晓记录不一致，应优先避免穿帮。客户端可以要求玩家重新同步库存，或恢复到安全的奖励展示状态。

## 成就与图鉴过滤

任何获得型系统都不应直接监听“Steam 库存新增”并立即触发反馈。

成就、图鉴、任务、活动收集、稀有播报和红点系统应只读取“已揭晓库存”，而不是直接读取完整 Steam 库存。

例如玩家开盲盒时，Steam 已经返回一个传说道具，但开盒动画尚未播放完成。此时游戏不能弹出“首次获得传说道具”之类的提示，也不能提前点亮图鉴。

正确触发时机是最终奖励图标首次展示之后。

## UI 表现分层

盲盒系统需要支持不同游戏模式下的开盒表现。

玩家可以在桌宠模式下开盲盒，也可以在标准扑克游戏模式下开盲盒。两种模式的开盒入口、布局空间、动画尺度和信息呈现方式可能不同。

因此，系统需要将正式库存结果、开盒演出路径和 UI 表现分开。

Steam Inventory 负责正式库存结果。  
开盒演出负责根据最终品质生成盲盒升级过程。  
UI 表现负责在当前模式中播放动画、接收点击、展示奖励图标，并在流程结束后收起。

这样后续即使桌宠模式和标准游戏模式拥有不同动画，也不会影响 Steam 库存规则。

## 筹码与正式库存边界

如果筹码只保存在本地存档中，玩家可以通过修改本地数据获得大量筹码。

因此，Steam 版本中需要谨慎处理“本地筹码购买正式 Steam 库存盲盒”的设计。

可选方向包括：

1. 本地筹码只用于本地游戏循环和轻量消费。
2. 正式 Steam 库存盲盒通过 Steam 掉落、活动、购买、签到券或其他 Steam 可校验方式获得。
3. 如果仍允许筹码购买正式盲盒，需要接受单机游戏中玩家修改本地筹码的风险。

现阶段可以先保留本地筹码购买盲盒用于快速测试，但正式 Steam 方案需要重新确认经济闭环。

盲盒价格阶梯递增、按时间重置的设计可以继续保留为策划规则，但不作为当前第一阶段开发重点。

## 异常与配置要求

盲盒系统应在开发和测试阶段尽早暴露配置问题。

以下情况应视为配置错误或需要明确兜底：

1. 本地 Item 表找不到 Steam 返回物品对应的本地配置。
2. Steam 返回物品品质没有对应的开盒演出路径。
3. 盲盒配置了升级表演，但没有最大点击次数或自动收起时间。
4. 活动奖励配置为直接展示，但没有奖励展示 UI 配置。
5. Steam 库存中存在待揭晓物品，但本地待揭晓记录丢失。
6. 本地待揭晓记录存在，但 Steam 库存同步后找不到对应实例。

正式环境中的兜底策略需要优先遵守“不穿帮”原则。

## 后续需要补充的配置

为了让系统完整数据驱动，后续至少需要补充以下配置：

1. 发奖来源配置：来源 ID、名称、来源类型、展示方式、领取次数限制、是否使用升级表演。
2. Steam 对齐配置：Steam itemdef ID、Steam generator ID、Steam 箱子或券 ID、Steam 交换目标 ID。
3. 盲盒表现配置：最大点击次数、奖励展示自动收起时间、是否允许提前结束。
4. 开盒演出路径配置：最终品质、初始品质、每次点击升级幅度、路径权重、是否允许提前结束。
5. 物品获取类型配置：初始拥有、普通盲盒、活动奖励、已下架、调试专用。
6. 道具显示配置：本地资源、图标、品质、排序、装备类型、国家隐藏规则、安全替换规则。
7. 签到或顺序领奖配置：领取券、进度令牌、每格奖励、最后一格重复规则。

Item 表可以继续作为本地显示和装备配置表。Steam 版本中，它需要能与 Steam Inventory 的 itemdef 对齐。
