# 本地化系统设计

## 文档目的

本文档记录 `Lucky Dog Rise` 游戏内本地化系统的设计原则、维护边界、翻译表结构、Key 命名规范、安全模式文案规则，以及分阶段实施计划。

当前项目由主人和 AI 助手共同开发。本地化内容后续可能由低成本 AI 批量处理，因此方案优先考虑维护成本、低歧义、可校验、少误改，而不是追求最少代码量。

## 设计目标

- 游戏内 UI 支持英文和简体中文。
- 默认语言跟随系统语言；不支持的系统语言回退英文。
- 本地化机制预留繁体中文、日文等后续语言扩展能力。
- 玩家可见文本集中维护，避免散落在场景、C#、Luban JSON 中。
- 安全模式文案与普通文案共用同一套本地化系统。
- 低成本 AI 只处理翻译表，不直接修改场景、C# 或 Luban 业务数据。
- 提供工具校验，尽早发现缺 key、重复 key、缺翻译、孤儿文本等问题。

## 非目标

- 第一版不实现繁体中文。
- 第一版不迁移所有道具名。
- 第一版不处理 Steam 库存后台的多语言 itemdef 文案。
- 第一版不追求全项目所有 Debug 文本都完成本地化。
- 不在每个 Luban 表中维护多语言列，例如 `Name_EN`、`Name_CN`、`Name_JA`。

## 语言范围

第一版支持：

- `en`：英文。
- `zh_CN`：简体中文。

后续预留：

- `zh_TW`：繁体中文，采用台湾用语风格，例如 `滑鼠`。
- `ja`：日文。

语言选择规则：

- 默认读取系统语言。
- 系统语言为简体中文时使用 `zh_CN`。
- 系统语言不在支持列表中时使用 `en`。
- 设置面板提供语言下拉选项，玩家手动选择后写入 `user://settings.cfg`。

## 字体策略

第一版继续使用当前项目字体：

- 英文：Inter。
- 中文：Xiaomi 中文字体。

繁体中文第一版不开放，因为当前字体资源未确认繁体中文字形覆盖范围。后续加入 `zh_TW` 前，需要单独检查字体覆盖和 UI 文本宽度。

## 翻译表

第一版使用单一 CSV：

```text
lucky-dog-rise/Data/Localization/LocalizationText.csv
```

第一版不拆分多个 CSV，原因是文本量尚小，单文件更不容易产生“这个 key 应该放哪个表”的维护歧义。

后续文本量明显变大时，可以拆分为：

```text
Data/Localization/ui.csv
Data/Localization/gameplay.csv
Data/Localization/items.csv
Data/Localization/debug.csv
```

拆分前必须先有校验工具，保证所有 CSV 中的 key 全局唯一。

## CSV 结构

CSV 第一列为 key，后续列为语言代码。

```csv
keys,en,zh_CN
Common_Quit,Quit,退出
Settings_Language,Language,语言
InfoPanel_GoodLuck,Good Luck!,祝你好运！
Paytable_RoyalFlush,Royal Flush,皇家同花顺
Paytable_RoyalFlush_Safe,Royal Feast,皇家盛宴
```

维护规则：

- `keys` 列不可为空。
- key 必须全局唯一。
- 第一版 `en` 和 `zh_CN` 都必须填写。
- 不支持语言的列不要提前加入，避免空列长期堆积。
- 翻译表只放玩家可见文本，不放策划备注、资源路径、数值规则。
- 若文本包含动态数字，使用占位符，不在代码中拼接语序。

示例：

```csv
keys,en,zh_CN
InfoPanel_YouWin,You win {0},赢得 {0}
BlindBox_AutoClaimingInSeconds,Auto-claiming in {0}s,{0} 秒后自动领取
```

## Key 命名规范

Key 采用“分类 + 语义 + 变体”的形式：

```text
Category_SemanticName_Variant
```

各部分规则：

- 使用英文。
- 使用大驼峰。
- 使用下划线分隔层级。
- 不使用空格、标点、中文、拼音。
- 不使用全大写蛇形命名，除非既有项目命名已经固定。
- key 必须是合法 C# 标识符，便于生成 `L10nKey.cs` 常量。

推荐示例：

```text
Common_Quit
Common_Cancel
Common_Confirm
Settings_Title
Settings_Language
Settings_Audio
InfoPanel_Chips
InfoPanel_Open
InfoPanel_GoodLuck
InfoPanel_YouWin
Paytable_RoyalFlush
Paytable_RoyalFlush_Safe
BlindBox_TapToPowerUp
BlindBox_TapToClaim
BlindBox_AutoClaimingInSeconds
```

不推荐示例：

```text
UI_BTN_QUIT
ui.button.quit
退出按钮
btn_quit
Quit
```

## Key 分类

建议分类如下：

- `Common`：通用按钮、确认、取消、关闭等。
- `Settings`：设置面板文本。
- `Wardrobe`：装扮面板文本。
- `Debug`：Debug 页玩家可见文本。
- `InfoPanel`：扑克信息面板文本。
- `Paytable`：牌型和赔率名称。
- `BlindBox`：盲盒提示、开奖、领奖文案。
- `Desktop`：桌宠模式计数器、提示气泡等。
- `Save`：存档、重置存档、损坏存档提示。
- `Item`：道具显示名，第一版暂不迁移。

## 安全模式文案

安全模式不是一门语言，而是文案变体。

安全模式 key 使用 `_Safe` 后缀：

```text
InfoPanel_Chips
InfoPanel_Chips_Safe
Paytable_RoyalFlush
Paytable_RoyalFlush_Safe
```

运行时请求普通 key：

```csharp
L10n.Tr(L10nKey.InfoPanel_Chips)
```

如果安全模式开启，本地化封装优先查询：

```text
InfoPanel_Chips_Safe
```

若 `_Safe` key 不存在，则回退普通 key：

```text
InfoPanel_Chips
```

这样只有敏感词或博彩感强的表达需要安全模式版本，普通文本不需要重复维护。

安全模式优先替换：

- `Chips` 等筹码相关表达。
- `Bet` 等下注相关表达。
- 扑克牌型中博彩感较强的表达。
- 盲盒或奖励中可能需要更温和表达的词。

安全模式不替换：

- 通用按钮，例如退出、确认、取消。
- 纯系统设置文本。
- Debug 内部文本。
- 不展示给玩家的策划备注。

第一版实现复用现有 `Streamer Safe Mode` 设置作为安全文案开关，不新增独立的 `Safe Text Mode` 设置。原因是第一版目标是减少概念数量，先验证普通文案与安全文案的切换链路。

## C# 封装

建议新增 `L10n` 静态类，职责如下：

- 初始化语言。
- 设置当前语言。
- 保存语言设置。
- 读取安全模式状态。
- 提供普通翻译方法。
- 提供格式化翻译方法。
- 对 `_Safe` key 做优先查找和回退。

建议接口：

```csharp
public static class L10n
{
    public static string CurrentLocale { get; }
    public static void ApplySavedOrSystemLocale();
    public static void SetLocale(string locale);
    public static string Tr(string key);
    public static string Format(string key, params object[] args);
}
```

建议生成 `L10nKey.cs`：

```csharp
public static class L10nKey
{
    public const string Common_Quit = nameof(Common_Quit);
    public const string Settings_Language = nameof(Settings_Language);
    public const string Paytable_RoyalFlush = nameof(Paytable_RoyalFlush);
    public const string Paytable_RoyalFlush_Safe = nameof(Paytable_RoyalFlush_Safe);
}
```

`L10nKey.cs` 不手写维护，由 Python 工具从 CSV 生成。

## Godot 场景文本

Godot 控件的 `text`、`tooltip_text`、`placeholder_text` 可以直接写 key。

例如：

```text
text = "Common_Quit"
tooltip_text = "Settings_Title"
placeholder_text = "Debug_FixedSeedPlaceholder"
```

Godot Translation 系统会按当前语言显示对应文本。

场景文本迁移规则：

- 玩家可见文本改成 key。
- 占位示例文本若不会在运行时展示，可以保留原文。
- Debug 页文本第一版可以部分保留英文，但后续应统一迁移。
- 数字和纯符号不进入翻译表，例如 `+0`、`88,888`、`12:00`。

## Luban 数据表策略

长期原则：

- Luban 业务表只保存稳定 key，不保存多语言文本。
- 玩家可见文本最终都在 CSV 中维护。
- 策划备注和内部名称可以继续留在 Luban 表中，不参与本地化。

推荐字段：

```text
NameKey
DescriptionKey
TitleKey
```

不推荐字段：

```text
NameEN
NameCN
NameTW
NameJA
SafeNameEN
SafeNameCN
```

第一版处理范围：

- `tbpaytable` 的玩家可见牌型名可以迁移为 key 或由代码映射 key。
- `tbitem` 的 `Name` 暂时不动，因为游戏内当前不展示道具名。
- Steam 库存后台显示名暂时单独处理，不纳入第一版游戏内本地化系统。

## Python 工具

允许为本地化流程编写 Python 工具。工具目录建议为：

```text
tools/localization/
```

建议工具：

```text
scan_strings.py
validate_strings.py
generate_l10n_keys.py
export_for_ai.py
import_ai_result.py
```

工具职责：

- 扫描 `.tscn` 和 `.cs` 中疑似硬编码玩家文本。
- 校验 CSV key 是否重复。
- 校验第一版语言列是否缺失翻译。
- 校验 `_Safe` key 是否符合命名规则。
- 校验 C# 和场景引用的 key 是否存在。
- 找出 CSV 中没人引用的孤儿 key。
- 从 CSV 生成 `L10nKey.cs`。
- 导出低成本 AI 可处理的待翻译清单。
- 合并 AI 翻译结果前执行格式校验。

低成本 AI 不应直接修改：

- `.tscn`
- `.cs`
- Luban JSON
- Godot 项目设置

## 第一阶段实施范围

第一阶段目标是跑通系统骨架，不追求全量文本迁移。

建议包含：

- 新建 `Data/Localization/LocalizationText.csv`。
- 在项目设置中注册 CSV 导入生成的翻译资源。
- 新增 `L10n` 封装。
- 新增语言设置持久化。
- 设置页增加语言下拉框。
- InfoPanel 的筹码、打开按钮、祝福语、胜利提示迁移到本地化。
- 盲盒舞台的主要提示文案迁移到本地化。
- 赔率表显示名迁移到本地化 key。
- 新增基础校验工具。

第一阶段暂不包含：

- 全量 SystemPanel 文本迁移。
- 全量 Debug 文本迁移。
- 道具名迁移。
- 繁体中文。
- 日文。
- Steam 库存后台多语言。

第一阶段实际范围固定为 `InfoPanel`、盲盒主要提示、赔率表和语言设置下拉框，不全量迁移 `SystemPanel`。

## 后续阶段

第二阶段：

- 迁移 SystemPanel 全量玩家可见文本。
- 迁移确认弹窗文本。
- 迁移 BossKey 桌宠提示文本。
- 补齐硬编码动态文本。

第三阶段：

- 迁移 Luban 玩家可见字段为 key。
- 处理 `tbpaytable`、`tbblindbox`、`tbtabgroup` 等表。
- 根据需要再处理 `tbitem`。

第四阶段：

- 加入繁体中文。
- 确认字体覆盖。
- 使用台湾用语风格进行人工审阅。

第五阶段：

- 加入日文等其他语言。
- 优化翻译表拆分。
- 接入 Steam 库存后台 itemdef 多语言导出流程。

## 验收标准

第一阶段完成后，应满足：

- 游戏启动时能根据系统语言选择 `en` 或 `zh_CN`。
- 设置页能手动切换语言，并写入设置文件。
- 重启后保持玩家选择的语言。
- 英文和简体中文都能正确显示。
- 不支持语言回退英文。
- 安全模式开启时优先显示 `_Safe` 文案。
- `_Safe` 文案缺失时回退普通文案。
- `dotnet build` 通过。
- 本地化校验工具通过。
