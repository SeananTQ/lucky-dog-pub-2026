## 概述

MVP版本基本算是结束了，虽然还有一些细节上的小问题。为了不做重复的无用功，我们开始开发桌宠需要的实际功能。

## 术语约定

为了避免后续沟通产生歧义，统一使用以下称呼：

| 术语 | 说明 |
|------|------|
| **宿主窗口** | 唯一的 OS 级窗口，无边框、透明背景、置顶、鼠标穿透。用 `DisplayServer` + Windows API 操作 |
| **游戏面板** | 宿主窗口内的扑克内容区域，始终可见，作为宿主窗口的”锚点”。内部所有元素响应拖拽 |
| **设置面板** | 宿主窗口内的 UI 控件，从游戏面板边缘弹出，智能定位。不响应拖拽 |
| **商店面板** | 未来的功能面板，同设置面板的定位方式 |
| ~**面板** | 设置面板、商店面板等的统称 |

关系：**宿主窗口 > { 游戏面板, 设置面板, 商店面板, ... }**。面板不是独立 OS 窗口，而是宿主窗口场景树中的 Control 节点。

要实现的功能：一个**智能浮层设置面板**。


## 核心目标

- 宿主窗口：无边框、透明背景、置顶，用户可以拖拽移动
- 游戏面板内有一个”设置”按钮
- 点击按钮后，在游戏面板**外部**（紧贴游戏面板的某一边）显示一个设置面板
- 设置面板里只有一个功能：音效开关（开启/关闭，状态保存到本地）
- 这个面板**看起来像是一个独立窗口**，但它**不是真正的独立系统窗口**，而是宿主窗口内的 UI 控件

## 面板的智能定位规则（重点）

面板不能出现在游戏面板内部，必须出现在外部。

定位逻辑如下：

首先获取屏幕总尺寸（例如 2560x1440）和游戏面板在屏幕上的矩形区域（宿主窗口位置 + 游戏面板在宿主窗口内的偏移 + 大小）。

默认情况下，设置面板优先显示在游戏面板的左侧。

如果左侧空间不足以完整显示面板（即游戏面板左边缘距离屏幕左边小于面板宽度），则尝试右侧。

如果右侧空间也不足（游戏面板右边缘距离屏幕右边小于面板宽度），则尝试上方。

如果上方空间也不足（游戏面板上边缘距离屏幕上方小于面板高度），则尝试下方。

如果所有方向都无法完整显示，则将面板强制显示在屏幕可见区域内，并与游戏面板保持最小间距，确保不被裁剪。

规则优先级为：**左 > 右 > 上 > 下**

每次宿主窗口位置变化时（用户拖拽结束），重新计算面板位置并移动它。


## 面板的视觉与交互

面板需要有背景色（半透黑）和边框。

面板上需要有一个关闭按钮（X），点击后关闭面板。

点击游戏面板的”设置”按钮时，如果面板已打开，就关闭它；如果已关闭，就打开并重新计算位置。

面板不能拖拽移动，其位置由程序完全控制。


## 音效设置功能

设置面板内需要包含一个开关控件，可以是 CheckButton 或 Toggle，标签为“音效开/关”。

该开关的状态需要保存到 `user://settings.cfg` 文件中。

## 技术约束

宿主窗口必须是透明、无边框、置顶。

游戏面板所在区域响应拖拽移动宿主窗口；面板不响应拖拽。

面板使用 `Control` 或 `Panel` 实现，不需要使用 `Window` 节点。这一点后续可以再考虑。
