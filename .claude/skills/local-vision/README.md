# local-vision

## 为什么需要这个东西

游戏开发中有一个根本矛盾：**编程 AI 没有视觉能力，但项目里的美术资源是图片**。

传统做法靠策划当翻译——美术画完图，策划告诉程序"这张图是底牌、那张图是取消按钮"。但单人独立项目没有策划，云端视觉模型又有隐私和成本问题（尤其当你的素材库包含不便上传的内容时）。

这个项目的目的就是：**让 AI 能用本地模型"看见"图片，替代策划的翻译角色**。所有图片数据不出本地网络。

## 设计哲学

**一切围绕"图、文分离"展开。**

游戏里的"文本"和"图片"是两回事。文本（卡牌名、费用、技能描述）最终由引擎里的 Text/TextMeshPro 组件渲染，不需要导出为图片。而图片（底板、图标、背景）是纯视觉资源，需要 AI 识别后赋予标准化的名字。

这套工具只处理"图"的部分，不碰"文"。文本由工作流的另一个分支（JSON 元数据）处理。

**解耦的三个层次：**

1. **看图**（必须本地模型）→ Python 脚本调 LM Studio，每张图生成结构化描述
2. **分类整理** → AI 读描述文件，按口头要求分类，不需要看图
3. **搜索** → 搜描述文件关键词，不需要重新调模型

第一层是核心，也是唯一依赖本地模型的环节。第二、三层可以交给任何 AI 处理，因为数据已经从图片变成了文本。

## 核心设计决策

| 决策 | 理由 |
|------|------|
| **零外部依赖** | 只用 Python 标准库（urllib, json, base64），复制到任何项目直接跑，不需要 pip install |
| **配置与代码分离** | config.json 管理 API 地址、模型名、提示词映射，改配置不改代码 |
| **提示词用 .md 文件** | 不写在代码里，新增提示词只需新建 .md 文件 + 改一行 config |
| **JSON 自动展开** | 模型返回 JSON 时自动合并到顶层，避免"JSON 套 JSON"的尴尬 |
| **断点续传** | 已有 .json/.txt 的图片自动跳过，大文件夹可以分批跑 |
| **错误隔离** | 每张图独立调用，一张图出事不卡住后面的；错误写 .txt 不写 .json |
| **不硬编码模型名** | 留空自动取当前加载的模型，减少配置文件改动 |

## 什么场景好用，什么不好用

**好用：**
- UI 元素（按钮、底板、图标、面板）——轮廓清晰、颜色分明
- 游戏截图——布局分析、文本识别
- 像素素材——边缘干净、内容明确

**不好用：**
- 抽象剪影——模型无法从轮廓判断具体物体
- 复杂大场景——细节丢失，描述泛化
- 隐私敏感图——但好在数据不出本地

## 工程结构

```
local-vision/
├── README.md                          ← 本文件
├── SKILL.md                           ← 技能定义，给 Claude Code/Trae 用的
├── config.json                        ← 所有配置（API、模型、提示词映射）
├── scripts/
│   └── analyze_image.py               ← 核心脚本，零依赖
└── prompts/
    ├── system.md                      ← 系统提示词，设定模型行为基调
    ├── describe.md                    ← 通用描述：一句话说清图片内容
    ├── game_asset.md                  ← 游戏资产分析：用途、视觉特征、文字、位置
    └── game_asset_structured.md       ← 结构化 JSON 输出：含尺寸、锚点、坐标
```

## 工作方式

```
图片文件 → base64 编码 → POST 到本地 LM Studio
                            ↓
                   模型推理 + 生成文本
                            ↓
              判断返回内容是否为 JSON
              ├─ 是 → 展开到顶层 + 加入 file 字段 → 输出
              └─ 否 → 放入 description 字段 → 输出
```

每次调用独立，无上下文残留。一个文件夹几百张图也能分批跑完。

## 配置

```json
{
  "api_base": "http://192.168.31.185:1234/v1",
  "model": "",
  "max_tokens": 4096,
  "batch_delay": 200,
  "batch_limit": 30,
  "system_prompt": "prompts/system.md",
  "prompts": {
    "default": "prompts/describe.md",
    "game_asset": "prompts/game_asset.md",
    "structured": "prompts/game_asset_structured.md"
  }
}
```

`model` 留空时脚本自动取 LM Studio 当前加载的第一个模型。改配置不需要改代码。

## 这个项目不是什么

- 不是云端 API 的封装——它强制使用本地模型
- 不是 MCP server——绕过了那层中间件，直接调 API
- 不是万能看图工具——模型能力决定上限，本地小模型理解剪影图的效果有限
